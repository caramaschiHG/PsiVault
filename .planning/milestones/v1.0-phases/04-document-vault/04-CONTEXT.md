# Phase 4: Document Vault - Context

**Gathered:** 2026-03-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 4 enables the professional to generate real practice documents from predefined templates (pre-filled with patient, professional, and date context), store them securely with provenance metadata, and retrieve them later from the patient profile. Document types covered: declaration of attendance, receipt, anamnesis, psychological report, and consent/service contract. Finance tracking, search, reminders, and communication features are separate phases.

</domain>

<decisions>
## Implementation Decisions

### Document types and templates
- All 5 document types use the **same generation experience** — one consistent flow for declaration, receipt, anamnesis, report, and consent/service contract.
- All documents pre-fill the **base set**: patient full name, professional name, CRP number, and today's date.
- Each type also gets **type-specific pre-fill fields** beyond the common base:
  - Receipt: amount and payment method (from appointment context if available)
  - Declaration of attendance: session count and date range
  - Anamnesis: intake date
  - Psychological report and consent/service contract: mostly blank-start beyond the base (professional writes the specifics)
- After pre-fill, the professional can **freely edit the document content** in an editable rich text area before saving — the template provides the starting point, not a locked form.

### Document generation flow
- **Entry point**: "Novo documento" button/section on the patient profile page only. No global entry, no agenda-level shortcut in v1.
- **Generation UX**: Full-page editor following the established "focused task = full route" pattern from Phase 3. Route: `/patients/[id]/documents/new?type=[documentType]`. Pre-filled editable document, consistent with clinical note composition.
- **Post-save**: After saving, the professional returns to the patient profile page — same behavior as clinical note saving.
- **Signature gate**: If the professional has not configured signature assets in their profile, document generation is **blocked** with a clear message ("Documentos requerem assinatura configurada. Configure em Perfil.") and a direct link to profile settings. This was decided in Phase 1 and is enforced here.

### Vault access posture
- **Viewing saved documents**: Dedicated in-app view page (read-only). No re-auth required to view — professional is already MFA-authenticated. Download option available from the view page.
- **Mutability**: Documents are **editable after saving**, with a full audit trail on each edit (actor, timestamp, what changed). Preserves the current content as the authoritative version.
- **Deletion**: No deletion in v1. Documents can only be archived (hidden from active views). Consistent with Phase 1 vault posture — nothing is destroyed.
- **Provenance display**: The document view header shows: document type, patient name, generated by (professional name), and created at timestamp. Scannable metadata at the top of the view page.

### Document listing on patient profile
- Documents section lives on the patient profile page, below the ClinicalTimeline section (following the established vertical structure of identity → summary → quick actions → clinical timeline → documents).
- List entries show: document type label, creation date, and a view action. Minimal surface, no clinical content exposed in the list (SECU-05).

### Claude's Discretion
- Exact rich text editor component choice (e.g., Tiptap, Quill, or a plain textarea with formatting) — as long as it supports editable pre-filled content.
- Template prose wording for each of the 5 document types in Portuguese.
- Exact layout and visual treatment of the document view page.
- Audit event schema for document create, edit, and archive actions.
- How "archive" is surfaced UX-wise (soft hide vs. explicit archive state visible in a filtered view).

</decisions>

<specifics>
## Specific Ideas

- Document generation route pattern should mirror the clinical note route: `/patients/[patientId]/documents/new?type=receipt` — consistent with Phase 3's `/sessions/[appointmentId]/note`.
- The vault posture means documents are treated like clinical records: they cannot be casually deleted, they carry provenance, and they live inside the patient's context.
- Signature assets being required before generation was an explicit Phase 1 decision — this phase enforces it at the UX level.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/lib/clinical/store.ts` + `src/lib/clinical/repository.ts`: The in-memory singleton store pattern should be replicated as `src/lib/documents/store.ts` for Phase 4.
- `src/lib/audit/events.ts` + `src/lib/audit/repository.ts`: Document create, edit (with changed fields), and archive events should extend the existing `AuditEvent` contract with appropriate `type` strings.
- `src/lib/setup/profile.ts` (`getPracticeProfileSnapshot`): Primary source for professional name, CRP, and signature assets used in every document pre-fill.
- `src/lib/patients/model.ts` + `src/lib/patients/repository.ts`: Patient identity fields (fullName, socialName) are the source for patient pre-fill in all document types.
- `src/lib/appointments/model.ts` + repository: Receipt and declaration types can pull session count, dates, and appointment context from completed appointments for richer type-specific pre-fill.

### Established Patterns
- Workspace-scoped ownership (`workspaceId`) — all document records follow the same boundary as patients, appointments, and clinical notes.
- In-memory repository singleton via `globalThis.__psivault*__` — document store should follow this pattern (no DB in v1).
- Full-page routes for focused tasks: `/patients/[patientId]/documents/new` fits the established routing grammar.
- Server actions passed as props to client form components (Phase 3 pattern) — document form server actions should follow the same colocation approach.
- Privacy baseline (SECU-05): document type and creation date can appear in patient profile list; document content must not appear anywhere outside the dedicated document view/editor routes.
- `nullCoerce` helper (Phase 3): optional document fields should follow the same null-over-empty-string policy.

### Integration Points
- `src/app/(vault)/patients/[patientId]/page.tsx`: Documents section added below `ClinicalTimeline`. The page already loads patient, appointments, profile, and clinical notes — document repository will be loaded here too.
- `src/lib/setup/readiness.ts`: Signature asset readiness check should be usable as a gate before rendering the document generation form.
- Audit repository: document lifecycle events (created, edited, archived) plug into the existing fire-and-forget audit pattern from server actions.

</code_context>
