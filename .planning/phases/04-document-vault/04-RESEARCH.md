## User Constraints (from CONTEXT.md)

### Locked Decisions

**Document types and templates**
- All 5 document types use the same generation experience — one consistent flow for declaration, receipt, anamnesis, report, and consent/service contract.
- All documents pre-fill the base set: patient full name, professional name, CRP number, and today's date.
- Each type also gets type-specific pre-fill fields beyond the common base:
  - Receipt: amount and payment method (from appointment context if available)
  - Declaration of attendance: session count and date range
  - Anamnesis: intake date
  - Psychological report and consent/service contract: mostly blank-start beyond the base
- After pre-fill, the professional can freely edit the document content in an editable rich text area before saving — the template provides the starting point, not a locked form.

**Document generation flow**
- Entry point: "Novo documento" button/section on the patient profile page only. No global entry, no agenda-level shortcut in v1.
- Generation UX: Full-page editor following the established "focused task = full route" pattern from Phase 3. Route: `/patients/[id]/documents/new?type=[documentType]`. Pre-filled editable document, consistent with clinical note composition.
- Post-save: After saving, the professional returns to the patient profile page — same behavior as clinical note saving.
- Signature gate: If the professional has not configured signature assets in their profile, document generation is blocked with a clear message ("Documentos requerem assinatura configurada. Configure em Perfil.") and a direct link to profile settings.

**Vault access posture**
- Viewing saved documents: Dedicated in-app view page (read-only). No re-auth required to view. Download option available from the view page.
- Mutability: Documents are editable after saving, with a full audit trail on each edit (actor, timestamp, what changed). Preserves the current content as the authoritative version.
- Deletion: No deletion in v1. Documents can only be archived (hidden from active views).
- Provenance display: Document view header shows document type, patient name, generated by, and created at timestamp.

**Document listing on patient profile**
- Documents section lives below the ClinicalTimeline section on the patient profile page.
- List entries show: document type label, creation date, and a view action. No clinical content in the list (SECU-05).

### Claude's Discretion
- Exact rich text editor component choice (e.g., Tiptap, Quill, or a plain textarea with formatting) — as long as it supports editable pre-filled content.
- Template prose wording for each of the 5 document types in Portuguese.
- Exact layout and visual treatment of the document view page.
- Audit event schema for document create, edit, and archive actions.
- How "archive" is surfaced UX-wise (soft hide vs. explicit archive state visible in a filtered view).

### Deferred Ideas (OUT OF SCOPE)
None — discussion stayed within Phase 4 scope.

## Summary

Phase 4 builds on the identical structural layer established in Phases 1-3. The codebase already has the complete pattern: in-memory singleton store via `globalThis.__psivault*__`, workspace-scoped model + repository, server actions passed as props to client form components, fire-and-forget audit via `createAuditEvent`, and `getPracticeProfileSnapshot` for professional identity. Phase 4 adds a sixth domain (`documents`) that replicates this pattern with two additions: a template engine that builds pre-filled prose per document type, and an archive flag (mirroring patient soft-archive semantics).

The "rich text editor" decision is Claude's discretion. Given the project's current implementation style (inline styles, no CSS framework, no external component library), a large textarea with `defaultValue` set to the pre-filled template prose is the lowest-friction approach and directly mirrors the `NoteComposerForm` pattern. Tiptap or similar editors add bundle weight and React state complexity without clear benefit at v1 scale — the recommendation is a styled `<textarea>` that satisfies the CONTEXT.md requirement ("supports editable pre-filled content").

The signature gate is an enforcement of an existing Phase 1 contract. `buildSetupReadiness()` already evaluates `signatureAsset` completeness. The page server component checks this before rendering the form and renders a blocking notice with a link to `/setup/profile` when the signature is absent.

**Primary recommendation:** Model `PracticeDocument` after `ClinicalNote`; build a `buildDocumentContent(type, context)` pure function for templates; replicate the `store/repository/model/audit` file quadruplet under `src/lib/documents/`; add a `DocumentsSection` component to the patient profile page below `ClinicalTimeline`; add two new routes: the composer at `/patients/[patientId]/documents/new` and the view at `/patients/[patientId]/documents/[documentId]`.

### Pattern 1: PracticeDocument Model (mirrors ClinicalNote)
**What:** Workspace-scoped document aggregate with provenance fields and an `archivedAt` soft-delete field.
**When to use:** Every document creation, update, and archive operation.
**Example:**
```typescript
// Source: mirrors src/lib/clinical/model.ts
export type DocumentType =
  | 'declaration_of_attendance'
  | 'receipt'
  | 'anamnesis'
  | 'psychological_report'
  | 'consent_and_service_contract';

export interface PracticeDocument {
  id: string;
  workspaceId: string;
  patientId: string;

  type: DocumentType;
  content: string;             // The editable text body (pre-filled then edited)

  // Provenance (DOCS-08)
  createdByAccountId: string;
  createdByName: string;       // Snapshot of professional name at creation time
  createdAt: Date;
  updatedAt: Date;
  editedAt: Date | null;       // null on first creation; set on every update

  // Soft archive (vault posture — no deletion in v1)
  archivedAt: Date | null;
  archivedByAccountId: string | null;
}
```

### Pattern 2: Template Factory (pure function)
**What:** A pure function that builds the initial document content string given the document type and a pre-fill context object. No side effects. The string is passed as `defaultValue` to the textarea.
**When to use:** Called server-side in the `page.tsx` of the composer route, before rendering the form.
**Example:**
```typescript
// Source: project-specific, mirrors established pure-function helpers
export interface DocumentPreFillContext {
  patientFullName: string;
  professionalName: string;
  crp: string;
  todayLabel: string;           // Formatted Portuguese date string
  // Type-specific (null when not applicable)
  sessionCount?: number | null;
  sessionDateRange?: string | null;   // "DD/MM/YYYY a DD/MM/YYYY"
  intakeDate?: string | null;
  amountLabel?: string | null;        // "R$ 180,00" stub for v1
  paymentMethod?: string | null;
}

export function buildDocumentContent(
  type: DocumentType,
  context: DocumentPreFillContext,
): string {
  switch (type) {
    case 'declaration_of_attendance':
      return buildDeclarationContent(context);
    case 'receipt':
      return buildReceiptContent(context);
    // ... etc.
  }
}
```

### Pattern 3: Repository (mirrors ClinicalNoteRepository)
**What:** In-memory Map keyed by document ID; same workspace-scoping contract.
**When to use:** All document persistence operations.
**Example:**
```typescript
// Source: mirrors src/lib/clinical/repository.ts
export interface PracticeDocumentRepository {
  save(doc: PracticeDocument): PracticeDocument;
  findById(id: string, workspaceId: string): PracticeDocument | null;
  listByPatient(patientId: string, workspaceId: string): PracticeDocument[];
  // Returns active (non-archived) docs only
  listActiveByPatient(patientId: string, workspaceId: string): PracticeDocument[];
}
```

### Pattern 4: Signature Gate (server-side guard)
**What:** The composer `page.tsx` checks `signatureAsset` from the practice profile before rendering the form. If absent, renders a blocking notice instead.
**When to use:** At the top of the `new/page.tsx` server component, before loading template context.
**Example:**
```typescript
// Source: mirrors guard pattern from src/app/(vault)/sessions/[appointmentId]/note/page.tsx
const profile = getPracticeProfileSnapshot(ACCOUNT_ID, WORKSPACE_ID);
if (!profile.signatureAsset) {
  return <SignatureRequiredNotice />;
  // Or render the notice inline — either way, the form is not shown
}
```

### Pattern 5: Audit Events for Documents
**What:** Three event types following the `domain.entity.action` naming convention established in `clinical/audit.ts`.
**When to use:** In server actions after each successful create, update, or archive.
**Example:**
```typescript
// Source: mirrors src/lib/clinical/audit.ts
export type DocumentAuditEventType =
  | 'document.created'
  | 'document.updated'
  | 'document.archived';

// SECU-05: metadata contains only documentId and documentType — never document content
```

### Anti-Patterns to Avoid
- **Exposing document content in the list view:** The `DocumentsSection` component must show only `type label + createdAt + view link`. Never render `doc.content` outside the dedicated view/editor routes.
- **Storing content in audit metadata:** Mirrors the SECU-05 constraint from clinical notes. Document content must never appear in audit event metadata.
- **Deleting documents:** No delete action in v1. The only removal is `archivedAt` soft-archive. All server action guards should reject delete-intent form submissions.
- **Re-deriving template text from the stored document:** Store `content` as the authoritative text after the user edits. Do not re-run the template factory on load of the view page; use the stored `content`.
- **Global document entry point:** The `new/page.tsx` must redirect or 404 if accessed without a valid `patientId`. No document can exist without a patient owner.

## Common Pitfalls

### Pitfall 1: Archive vs. Active in List
**What goes wrong:** `listByPatient()` returns archived documents, polluting the active documents section on the patient profile.
**Why it happens:** Forgetting to filter by `archivedAt === null` when listing.
**How to avoid:** Provide `listActiveByPatient()` as the primary method; the profile page calls this. `listByPatient()` returns all (for internal/recovery use).
**Warning signs:** Documents don't disappear from the profile section after archive action is invoked.

### Pitfall 2: Document Type Not Validated in Server Action
**What goes wrong:** The `?type=` URL query param accepts any string and gets stored as-is, corrupting the type discriminant.
**Why it happens:** Forgetting to validate the `type` param against the `DocumentType` union before calling `buildDocumentContent`.
**How to avoid:** Guard with an explicit set check at the top of the server action and the page.tsx: `const VALID_TYPES = new Set([...])`.
**Warning signs:** Type label on the view page shows an unrecognized string or undefined.

### Pitfall 3: Template Prose in a Client Component
**What goes wrong:** Moving `buildDocumentContent` to a client component exposes the template logic and the pre-fill context to the browser bundle.
**Why it happens:** Trying to use React `useState` to hold the pre-filled content.
**How to avoid:** Build the pre-filled content server-side in `page.tsx` and pass it as a prop (`defaultContent: string`) to the form client component. The client component only manages `isDirty` state, exactly like `NoteComposerForm`.
**Warning signs:** The `buildDocumentContent` import appears in a `"use client"` file.

### Pitfall 4: SECU-05 Violation in Document List
**What goes wrong:** The `DocumentsSection` component renders the first N characters of `doc.content` as a preview, leaking clinical content to the patient profile surface.
**Why it happens:** Copy-paste from a "preview" design pattern without checking the security constraint.
**How to avoid:** The list entry only renders: type label, formatted `createdAt`, and a `<Link>` to the view page. Never render `doc.content` in list context.
**Warning signs:** Document content appears on the patient profile page outside the `/documents/[documentId]` route.

### Pitfall 5: Receipt Pre-fill Blocked on Phase 5 Finance Domain
**What goes wrong:** Receipt generation fails or shows empty amount because the finance domain (Phase 5) hasn't been built yet.
**Why it happens:** Attempting to read appointment price from a not-yet-existing field.
**How to avoid:** Receipt pre-fill in Phase 4 uses empty strings / placeholder text for amount and payment method. The CONTEXT.md states "from appointment context if available" — since Phase 5 has not shipped, these values default to empty and the professional fills them manually.
**Warning signs:** Compilation errors referencing non-existent `priceInCents` on `Appointment`.

## Open Questions

1. **Receipt amount pre-fill before Phase 5**
   - What we know: Phase 5 adds `priceInCents` to the finance domain; Phase 4 ships before Phase 5.
   - What's unclear: Whether to use empty string, `"(valor a preencher)"`, or skip the field in the pre-fill template.
   - Recommendation: Use a visible placeholder like `"R$ ________"` in the receipt template prose so the professional knows to fill it in manually. This is a template content decision (Claude's discretion).

2. **Archive UX surface**
   - What we know: Archive is a locked feature; UX treatment is Claude's discretion.
   - What's unclear: Whether the archive action is accessible from the list entry or only from the document view page.
   - Recommendation: Archive action lives on the document view page only (not in the list), consistent with the vault posture that discourages casual removal.

3. **Download implementation**
   - What we know: Download option must be available from the view page (CONTEXT.md locked decision).
   - What's unclear: Whether this is a browser-native download (data: URI), a server route response, or simply `window.print()` to PDF.
   - Recommendation: Use `window.print()` or a `<a href="..." download>` pointing to a route that returns the content as `text/plain` or `text/html`. A dedicated download route (`/patients/[patientId]/documents/[documentId]/download`) returning the document body as a `text/plain` response is the simplest secure approach for v1.
