Padrão CORRETO para actions que terminam com redirect():
```typescript
export async function someAction(formData: FormData) {
  try {
    // toda a lógica da action — guards, repo calls, audit.append
    await repo.save(entity);
    audit.append(...);
  } catch (err) {
    console.error("[someAction]", err);
    return { ok: false, error: "Algo deu errado. Tente novamente." };
  }
  redirect("/destino"); // FORA do try — redirect não será capturado
}
```

Padrão CORRETO para actions que retornam { ok: boolean } (sem redirect):
```typescript
export async function someAction(input: {...}) {
  try {
    // lógica
    return { ok: true };
  } catch (err) {
    console.error("[someAction]", err);
    return { ok: false, error: "Algo deu errado. Tente novamente." };
  }
}
```

PITFALL CRÍTICO: redirect() do Next.js lança NEXT_REDIRECT internamente.
Se redirect() estiver DENTRO do try, será capturado e engolido pelo catch.
Solução: sempre mover redirect() para APÓS o bloco try/catch.

Regras de log:
- Logar: nome da action entre colchetes + o objeto err
- NUNCA logar: conteúdo de campos clínicos, financeiros, ou dados do paciente
- Exemplo correto: console.error("[createNoteAction]", err)
- Exemplo proibido: console.error("[createNoteAction] failed for patient:", patientId, noteData)

Funções que retornam implicitamente (guard "if (!existing) return"):
- Manter os guards DENTRO do try para capturar falhas de repo.findById
- O "return;" sem valor nesses guards é aceitável — não precisam retornar { ok: false }
  (são guards de "not found", não erros de infraestrutura)

Actions que lançam Error explicitamente (ex: conflito de agendamento):
- Substituir throw new Error(...) por return { ok: false, error: "..." } pt-BR dentro do try
  OU manter throw e deixar o catch capturar — opção mais simples
- Preferir manter throw dentro do try (catch captura e retorna { ok: false })
